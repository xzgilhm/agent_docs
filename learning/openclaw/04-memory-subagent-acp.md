# OpenClaw Memory、Skill、Repo 文档与 Prompt 分层

这一节不是只讲 `Memory` 本身，而是讲一个更重要的工程问题：

> **怎样让 agent 长期稳定地按预期做事，而不是偶尔想起来。**

很多人第一次接触 agent 时，会把“记忆”理解成万能层，仿佛只要把要求存进去，系统以后就会稳定遵守。

这不对。

更准确的理解是：

- `Memory` 更像**事实层 / 检索层**
- `Skill` 更像**方法层 / 作战手册**
- `Repo 文档` 更像**项目约束层**
- `Prompt` 更像**本次执行层**

如果把这四层混在一起，agent 就会表现出一种很常见的问题：

- 有时能想起来
- 有时又忘
- 有时看起来知道，但做不到位
- 有时被当前上下文冲掉

所以真正稳定的做法，不是“把东西都塞进 memory”，而是**按层放置信息**。

---

## 一句话先记住

- `Memory`：帮助想起来
- `Skill`：帮助知道怎么做
- `Repo 文档`：帮助进入项目就看到必须遵守的规则
- `Prompt`：帮助这一轮执行必须命中要求

可以把它记成一句口令：

> **事实进 memory，流程进 skill，约束进文档，必做项进 prompt。**

---

## 1. Memory：事实层，不是执行保证层

### 它适合放什么

`Memory` 最适合放的是稳定事实：

- 用户偏好
- 长期目标
- 环境事实
- 长期背景约束
- 需要跨会话记住的上下文

例如：

- Ole 偏好先给结果、少铺垫
- 当前学习优先级是“告警/日志/巡检助手”
- 招聘筛选核心是“单位稳定 + 运维相关”

这些信息的共同特点是：

- 跨很多任务都可能有用
- 不是某一轮执行的具体步骤
- 更像“这个人/这个长期方向是什么”

### 为什么它不适合承载关键流程

因为 memory 的本质更像：

- 可检索的长期记忆
- 被压缩过的提炼事实
- 在上下文预算内择优注入的一部分信息

这意味着它有天然不稳定性：

1. **不一定检索得到**
   - 关键词不对，可能没命中
   - 历史记录没有清楚表达，也可能没命中

2. **检索到了不一定注入**
   - 相关性排序、上下文预算、当前任务优先级都会影响注入

3. **注入了不一定主导执行**
   - 当前任务 prompt、更高优先级指令、长上下文里的竞争信息都可能把它盖过去

所以你可以把 memory 理解成：

> **它提高“想起来”的概率，但不提供“做到”的强保证。**

---

## 2. Skill：方法层，把经验固化成可主动加载的流程

如果某件事不是“一个事实”，而是“应该怎么做”，那通常更适合进 `Skill`。

### Skill 适合放什么

- 标准操作流程
- 工具路线
- 常见坑
- 验证步骤
- 场景触发条件
- 失败后的替代路径

比如：

- 做招聘站点探测时，先看官方站，再看聚合站
- 用浏览器自动化前先确认是否有 API / 网络请求可直接利用
- 教学回答后不要停在解释，要补一小步练习或提问

### 为什么 skill 比 memory 更适合流程

因为 skill 的使用方式是：

- 在任务开始前**主动加载**
- 以文档形式展开完整流程
- 往往包含触发条件、步骤、注意事项、验证方式

这和 memory 完全不是一个粒度。

可以这么类比：

- `Memory` 像“我记得你大概偏好什么”
- `Skill` 像“这类任务我们有一份现成 runbook”

所以如果你发现某条信息长这样：

- “遇到 X 任务，应该先做 A，再做 B，最后验证 C”

它大概率不是 memory，而是 skill。

## 3. Repo 文档：项目约束层

有些要求既不是“用户偏好”，也不是“通用方法”，而是：

> **只要进入这个项目，就必须遵守。**

这类内容最适合放在 repo 文档或本地工作区文档里，比如：

- `AGENTS.md`
- `README.md`
- `docs/runbook.md`
- `architecture.md`
- `ADR`（架构决策记录）

### 它适合放什么

- 项目目标
- 目录规范
- 接口约定
- 输出格式
- 安全边界
- 当前项目里的角色分工
- 特定 agent 的工作契约

### 为什么它重要

因为 repo 文档有一个很大的稳定性优势：

> **只要进入这个项目，就能读到。**

它不依赖历史检索，也不依赖我“突然想起来”。

比如：

- 某个告警助手要求所有摘要必须输出固定四段
- 某个项目规定所有分析结果必须先写报告再回消息
- 某个工作区规定不能直接删文件，只能走 trash

这种“当前项目铁规则”，写进 repo 文档，比只放 memory 强得多。

---

## 4. Prompt：本次执行层

还有一层最硬的约束，叫“本次执行层”。

这就是 prompt，尤其是：

- cron prompt
- autonomous task prompt
- 一次性任务的明确指令

### 它适合放什么

- 这次一定要做的动作
- 输出格式
- 成功标准
- 不允许省略的步骤
- 找不到时如何回复

比如每日岗位归档任务，prompt 就应该显式写：

- 抓取稳定单位里的运维相关岗位
- 优先官方招聘渠道
- 输出单位、岗位、地点、链接、判断理由
- 若无新增，也要明确回报“今日无新增”

### 为什么这层必须自包含

因为 cron/job 运行时最怕的一件事就是：

> **默认它这次什么都不记得。**

所以像下面这种 prompt 都不稳：

- “按之前的方式继续做”
- “参考上次要求”
- “沿用既有标准”

这类写法的问题是：

- “之前”不一定能被找回
- “上次”不一定会注入
- “既有标准”不一定在当前上下文最显眼

因此好的自动任务 prompt 应该满足：

> **即使失忆，只看这一段也能把事做对。**

## 5. 为什么很多 agent 会“好像记住了，但做得不稳”

因为它们把四层信息混在一起了。

### 常见错误 1：把流程写进 memory

例如：

- “以后每次做招聘任务都先探官方 API，再抓页面，再导出表格”

这看起来像“记忆”，其实是流程。

问题是：

- memory 可能只保留成一句模糊总结
- 之后检索命中也未必完整
- 真到执行时，步骤细节很容易丢

更稳的做法是写成 skill 或 runbook。

### 常见错误 2：把关键偏好只留在聊天里

例如：

- “回答完后继续带着我学”

如果只出现在某次聊天里，那就是最脆弱的形态。

更稳的是：

- 把“用户期待连续引导式学习”记进 memory
- 如果形成固定教学策略，再固化进教学 skill

### 常见错误 3：cron prompt 过度依赖历史

这也是最常见的自动化故障来源。

只要 prompt 里出现：

- 按之前方式
- 参考以前标准
- 延续既有习惯

就说明它对历史检索有依赖。

而对自动任务来说，依赖历史越多，稳定性通常越差。

## 6. 一个简单实用的判断法

当你准备存一条信息时，先问自己：

### 问题 1：它是在描述“是什么”，还是“怎么做”？

- “是什么” → 倾向 `Memory`
- “怎么做” → 倾向 `Skill`

### 问题 2：它是长期通用，还是项目专属？

- 长期通用 → 倾向 `Memory` / `Skill`
- 项目专属 → 倾向 `Repo 文档`

### 问题 3：它是不是本次执行绝不能漏？

- 是 → 必须进入 `Prompt`

可以压缩成一个四选一判断：

- **事实** → memory
- **流程** → skill
- **约束** → repo 文档
- **必做项** → prompt

## 7. 用告警/日志/巡检助手举例

这套分层对你后面要做的“告警/日志/巡检助手”非常重要。

### Memory 层可能放什么

- 用户更关注高优先级、可执行的告警摘要
- 对噪声告警容忍度低
- 当前主要场景是运维/基础设施排障

### Skill 层可能放什么

- 告警归类流程：基础设施、应用、网络、容量、安全
- 降噪流程：去重、聚合、抑制、升级
- 初步根因分析套路：先看时间线、再看依赖、再看指标、再看变更
- 输出建议动作模板

### Repo 文档层可能放什么

- 告警输入 schema
- 摘要输出字段规范
- 优先级定义
- 升级规则
- 值班流转规则

### Prompt 层可能放什么

- 这次输入的告警数据在哪
- 这次输出必须包含哪些字段
- 这次是否要求产出 markdown / JSON / message
- 找不到根因时应该如何明确表达不确定性

你会发现：

> 这不是“记忆设计”一个点的问题，而是**整个 agent 控制面的分层设计**。

---

## 8. 最后给你一个非常实用的结论

如果一条信息“忘了会出事故”，那它就不该只存在于 memory。

换句话说：

- 只是为了更懂你 → 放 memory
- 为了重复做好某类任务 → 放 skill
- 为了这个项目谁来都不出错 → 写进 repo 文档
- 为了这一轮绝不漏步骤 → 写进 prompt

所以真正靠谱的 agent，不是 memory 特别大，
而是：

> **分层清楚，关键要求放在正确位置。**

---

## 小结

把这一节压成四句话：

1. `Memory` 帮助想起来，但不保证做到
2. `Skill` 用来固化“怎么做”的方法
3. `Repo 文档` 用来承载项目内必须遵守的约束
4. `Prompt` 用来保证本次执行绝不漏关键要求

如果以后你只记一句，就记这句：

> **越怕忘，越不能只放 memory。**
