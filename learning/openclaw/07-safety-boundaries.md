# OpenClaw 权限与安全边界

这一节讲 OpenClaw 为什么不是“有工具就乱用”，而是需要权限、边界和流程控制。

## 一句话先记住

> Tool 让 agent 能做事，权限与安全边界决定 agent 在什么条件下可以做、应该做到哪一步。

---

## 1. 为什么权限与安全边界重要

OpenClaw 不是纯聊天系统，而是能：

- 读文件
- 写文件
- 执行命令
- 访问网络
- 推送到 GitHub
- 操作会话和子任务

这意味着它有真实执行力。

所以问题不只是：

- agent 会不会做

还包括：

- agent 该不该做
- 什么时候要先确认
- 什么动作风险更高
- 如何避免误操作和越权

---

## 2. 权限控制在防什么

主要在防这几类问题：

- **破坏性操作**：误删文件、误改配置、误停服务
- **越权操作**：在不该外发时对外发送内容
- **隐私泄露**：把不该暴露的内容发到群聊或外部服务
- **错误自动化**：在用户还没确认前就直接做高风险动作
- **工具滥用**：把 shell、网络、外部写操作用在不合适的地方

---

## 3. OpenClaw 里的安全思路

OpenClaw 的安全思路不是“完全不能做”，而是：

- 低风险动作尽量直接做
- 高风险动作要更谨慎
- 有些动作需要明确确认
- 不同场景下上下文和记忆要隔离
- 把权限、流程、skill、tool 一起考虑

所以它更像“有护栏的执行系统”。

---

## 4. 常见的风险分层

### 低风险

例如：

- 读文件
- 看状态
- 搜 memory
- 浏览文档
- 写学习文档到 wiki 工作区（未推送前）

通常这类动作可以直接做。

### 中风险

例如：

- 修改已有配置文件
- 长时间执行命令
- 对仓库进行结构性改动
- 拉起复杂后台任务

这类动作通常要更谨慎，必要时说明一下在做什么。

### 高风险

例如：

- 删除重要文件
- 停服务
- 改系统级配置
- 向外部发送敏感内容
- 使用高权限命令

这类动作往往不该静默执行，需要更明确的确认和审查。

---

## 5. 为什么“群聊”和“主会话”边界不同

因为 OpenClaw 可能同时出现在：

- 用户私聊
- 群聊
- 外部集成场景

主会话里可以看到的一些长期信息，不代表在群里也该说。

所以一个成熟 agent 要理解：

- 能看到，不代表能随便说
- 能访问，不代表能随便外发

这就是为什么 memory、会话上下文、外部消息要区分场景。

---

## 6. Skill 在安全里也有作用

Skill 不只是提效，也能约束行为。

比如一个好的 skill 会告诉 agent：

- 哪些步骤要先确认
- 哪些写操作要谨慎
- 哪些内容不要对外发
- 哪些情况下该停下来问用户

所以 skill 的价值不只是“更会做”，还有“更不会乱做”。

---

## 7. 一个成熟 agent 的安全习惯

你可以把它总结成四句：

- 先判断风险，再执行
- 能内部完成的，别急着对外
- 高风险动作先确认
- 有边界意识，不把访问权限等于表达权限

---

## 8. 这一节最该带走的理解

看完这一节，你至少应该记住：

- OpenClaw 的核心风险来自它有真实执行力
- 权限控制解决的是“该不该做”和“做到哪一步”
- skill 和安全不是分开的，skill 也能帮助约束行为
- 访问到信息不等于应该把它说出去
