# 06 - 权限、安全、审批与隔离边界

这一章开始讲一个 agent 系统里最容易被嘴上说重视、但工程上最容易做糊的部分：

- 权限
- 安全
- 审批
- 隔离边界

很多人一谈 agent 安全，第一反应就是：

- “别让它乱执行命令”
- “别让它乱发消息”
- “别让它权限太大”

这些都没错，但都还是偏表层。

真正从系统设计视角看，更关键的问题是：

> **系统怎么决定：它能做什么、不能做什么、什么时候自己做、什么时候必须拦住、以及出事时能把问题困在多小的范围里。**

这一章讲的不是道德说教，而是运行时边界设计。

---

## 1. 先记一句最关键的话

> **agent 安全的核心，不是单点禁止，而是“权限范围 + 执行审批 + 隔离半径”三件事一起成立。**

也就是说，安全不是只问：

- 能不能做

而是同时问：

- **默认能做什么**
- **什么情况要确认**
- **做坏了能影响多大范围**

如果这三件事没同时想清楚，系统通常会滑向两个极端：

- 要么太松：看起来很能干，但风险不可控
- 要么太死：看起来很安全，但几乎什么都干不了

一个可用的 agent 系统，追求的是：

> **在可控边界内尽量自动化，在高风险边界上明确收口。**

---

## 2. 为什么 agent 的安全问题和普通脚本不一样？

因为 agent 不是固定脚本。

普通脚本通常是：

- 输入较固定
- 流程较固定
- 能做的事相对可枚举

而 agent 的麻烦在于：

- 输入是开放的
- 规划路径不完全固定
- 它会根据上下文动态选工具、调子流程、访问外部资源

所以风险不只是“某个命令危险”，而是：

> **动态规划 + 外部工具 + 持久上下文 + 跨系统动作** 叠加后的组合风险。

也正因为这样，agent 的安全设计不能只靠：

- 一个黑名单
- 几句 prompt 警告
- 人工相信模型会听话

这些都不够稳。

---

## 3. 把安全问题拆成四层，会清楚很多

从工程角度，你可以先把 agent 安全拆成四层：

1. **能力边界**：系统理论上能调用哪些能力
2. **权限边界**：当前会话 / 当前 runtime 实际被授予哪些能力
3. **审批边界**：哪些动作必须先确认
4. **隔离边界**：即使出错，影响范围能否被限制

这四层不要混。

### 3.1 能力边界

能力边界回答的是：

> **这套系统总体上接了哪些刀。**

比如系统可能拥有：

- 读文件
- 写文件
- 执行 shell
- 发消息
- 浏览器自动化
- 调 GitHub / Gmail / 日历 / 云 API

如果某种能力根本没接入，风险就不存在。

所以第一步不是问“它会不会乱用”，而是先问：

> **这个能力有没有被接进系统。**

### 3.2 权限边界

即使系统总体支持某个能力，也不代表当前任务就该拿到它。

权限边界回答的是：

> **这次执行现场，实际开放了哪些能力。**

比如：

- 主 agent 可以读写文件
- 某个子 agent 只能读文件不能发消息
- 某个 ACP 会话能跑代码，但不能访问浏览器
- 某些线程能发 Telegram，但不能发邮件

这就是为什么“系统有这把刀”与“当前会话手上有这把刀”不是一回事。

### 3.3 审批边界

审批边界回答的是：

> **哪些动作不该默认自动通过。**

最典型的是：

- 外部发送
- 删除/覆盖重要数据
- 高成本 API 调用
- 对生产环境的修改
- 涉及隐私、财务、账号、公开发布的动作

这里的重点不是“永远不能做”，而是：

> **在风险足够高时，需要显式确认。**

### 3.4 隔离边界

这是很多人最容易忽略的一层。

隔离边界回答的是：

> **就算系统判断错了、模型犯傻了、工具异常了，损害能不能被困在局部。**

比如：

- 子 agent 拿不到主会话全部上下文
- 某个运行环境只能访问指定目录
- 某个工具只能写工作区，不能碰系统目录
- 某个 token 只允许只读，不允许写入

真正成熟的安全设计，靠的不是“绝不犯错”，而是：

> **犯错时也别一下炸穿整套系统。**

---

## 4. 为什么审批不是“麻烦用户一下”这么简单？

很多人会把审批理解成：

- 碰到危险动作就问一句

这太浅了。

审批真正解决的是两个问题：

1. **责任转移点在哪里**
2. **系统如何区分低风险自动化和高风险动作**

### 4.1 低风险动作应该尽量自动

例如：

- 读文件
- 查日志
- 搜索代码
- 本地生成草稿
- 在工作区里写可回滚文件

如果这些动作都要不停审批，系统就会变成残废。

### 4.2 高风险动作必须收口

例如：

- 对外发送消息
- 修改生产配置
- 删除关键文件
- 运行高破坏性命令
- 使用高权限 token 做不可逆操作

这些动作如果不收口，就不是“高效”，而是“迟早出事”。

所以审批不是为了形式感，而是为了在系统里画出一条明确线：

> **线内自动，线外确认。**

---

## 5. 为什么“不要只靠 prompt 保安全”？

这是最需要纠正的工程误区之一。

很多人会想：

- system prompt 里写清楚就行
- 告诉模型不要乱做事
- 让它注意安全边界

这些当然有价值，但只能算**软约束**。

软约束的问题是：

- 可能被上下文冲淡
- 可能被错误检索或错误装配覆盖
- 可能在复杂任务里被模型误判
- 不能替代 runtime 真正的权限控制

所以更稳的顺序应该是：

1. **先用 runtime / tool 权限做硬边界**
2. **再用审批规则做动作收口**
3. **最后用 prompt / skill 做行为提醒**

也就是说：

> **prompt 负责提醒，runtime 负责兜底。**

如果反过来，只靠 prompt 顶风险，系统迟早会漏。

---

## 6. 隔离为什么比“聪明”更重要？

很多人做 agent 时容易追求：

- 更聪明
- 更能推理
- 更会自主规划

这些当然重要。

但一旦 agent 真能做事，隔离的重要性会迅速上升。

因为越聪明、越能动手的系统，一旦边界做差，破坏力也越强。

### 6.1 常见隔离对象

你可以重点关注这些隔离面：

- **上下文隔离**：不同 session / subagent 之间不要无脑共享全部上下文
- **文件隔离**：限制可读写目录
- **进程隔离**：限制可执行命令和运行环境
- **网络隔离**：限制可访问目标和外部 API
- **凭证隔离**：不同能力用不同 token，不要一把万能钥匙
- **身份隔离**：不要让所有外发动作都以同一高权限身份执行

### 6.2 一个很实用的原则

> **宁可多做几层小隔离，也别指望一个总开关保护一切。**

因为总开关一旦配错，整套系统一起裸奔。

---

## 7. 从运行时视角，安全边界通常落在哪些地方？

如果你把 OpenClaw 这类系统拆开看，安全边界一般会落在这些节点：

### 7.1 工具暴露层

系统到底暴露哪些工具给模型，这是第一道闸。

### 7.2 工具参数校验层

就算允许调用某个工具，也要看参数是否合法、是否越界。

### 7.3 运行时审批层

当动作触发高风险规则时，要不要拦下来等确认。

### 7.4 会话 / 子代理作用域层

某个子任务是否继承全部能力，还是只拿最小权限。

### 7.5 外部系统凭证层

第三方 API token 的读写权限、作用域、过期策略是否合理。

你会发现，真正的安全从来不是一个点，而是：

> **多层小边界叠加出来的整体约束。**

---

## 8. 一个很常见的误判：把“安全”理解成“什么都别做”

这也是很多系统最后不好用的原因。

如果一个 agent：

- 什么都要问
- 什么都不能写
- 什么都不能运行
- 什么都不能联网

那它当然安全很多，但也基本失去了 agent 的价值。

所以更成熟的目标应该是：

- **对低风险内部动作放行**
- **对高风险外部动作审批**
- **对破坏性动作加硬限制**
- **对跨边界动作加强隔离和审计**

这才是可用性和安全性的平衡。

---

## 9. 运维视角下，排查安全问题时该怎么想？

当系统表现出这些症状时：

- 会说不会做
- 某些工具突然调不动
- 某个子 agent 权限怪怪的
- 某些消息发不出去
- 某些环境能跑，某些环境不能跑

不要只想“是不是 bug”。

要同时考虑它是不是边界设计在起作用。

更好的排查顺序通常是：

1. **能力有没有暴露**
2. **当前会话有没有权限**
3. **是否命中了审批门槛**
4. **是否被隔离策略拦住**
5. **日志里记录的是 hard deny、soft deny 还是参数错误**

这很重要，因为：

> **很多看起来像故障的问题，本质上其实是安全边界在正常工作。**

当然，反过来也成立：

- 如果你以为是安全策略在拦，结果实际上是 runtime 配坏了
- 那就会把真实故障误当成“预期限制”

所以安全也必须是可观察的，而不能黑箱。

---

## 10. 好的 agent 安全设计，至少该满足什么？

从工程角度，一个比较成熟的系统至少应满足：

1. **能力可枚举**：系统有什么刀，能说清
2. **权限可收缩**：不同 session / agent 能拿不同权限
3. **审批可解释**：为什么被拦、为什么需要确认，能解释
4. **隔离可验证**：文件、网络、凭证、上下文边界能验证
5. **动作可审计**：谁在什么时候调了什么工具，最好能追
6. **失败可控**：即使模型误判，也别直接造成大面积损害

如果这些做不到，所谓“安全”大概率只是心理安慰。

---

## 11. 这一章最该带走的四件事

### 第一件：安全不是一句“别乱来”

它至少包含：

- 权限范围
- 审批规则
- 隔离半径

### 第二件：能力、权限、审批、隔离要分开理解

它们不是同一个问题，只是一起作用。

### 第三件：不要只靠 prompt 保安全

prompt 是提醒，不是硬边界。
真正的兜底应该在 runtime、tool 权限和作用域控制里。

### 第四件：安全不是为了让 agent 什么都不能干

而是为了让它在低风险范围内高效自动化，在高风险范围上明确收口。

---

## 12. 用一句话收尾

> **agent 安全不是“让模型变乖”，而是把能力、权限、审批和隔离做成一套真实会生效的运行时边界。**

你把这层理解清楚了，后面再看后台任务、持久化执行、长期自动化，就不会只问“它能不能做”，而会先问“它应该在多大边界内做”。

看完这章，你至少应该能回答：

1. 为什么 agent 安全不能只靠 prompt？
2. 能力边界、权限边界、审批边界、隔离边界分别在解决什么问题？
3. 为什么审批不是“麻烦用户一下”，而是系统责任边界的一部分？
4. 为什么隔离设计比单纯要求模型“更聪明”更关键？

---

## 一句话小结

**安全不是让 agent 什么都别做，而是让它在清楚的权限、审批和隔离边界内做该做的事。**
