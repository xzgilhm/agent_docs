# Teaching Lenses

Use this file when a topic needs more than one teaching angle.

## Lens Selection

| User need | Primary lens | Secondary lens |
|---|---|---|
| "What is this?" | Concept | Map |
| "How do I use this?" | Practice | Concept |
| "Why is it broken?" | Troubleshooting | Practice |
| "How do I answer this in interview?" | Interview | Concept |
| Broad new topic | Map | Concept |
| Preparing for work | Practice | Troubleshooting |

## Lens Patterns

### Map Lens
Use when the user lacks orientation.

Include:
- what problem this topic solves
- where it sits in the stack or workflow
- 3-5 key components
- one sentence per component

### Concept Lens
Use when the user needs a correct mental model.

Include:
- definition in plain language
- why it exists
- what it is not
- one simple example
- one easy-to-confuse adjacent concept

### Practice Lens
Use when the user needs to operate the thing.

Include:
- where it appears in real work
- usual commands, files, dashboards, or workflows
- common defaults or conventions
- one realistic mini-case

### Troubleshooting Lens
Use when the user is stuck or the topic is failure-heavy.

Include:
- symptom → likely causes
- recommended debugging order
- what to check first vs later
- one anti-pattern to avoid

### Interview Lens
Use when the user needs concise articulation.

Include:
- 30-second answer
- 2-3 expected follow-up questions
- practical example that makes the answer stronger
- one nuance that separates beginner and intermediate answers

## Sequencing

Prefer these sequences:
- new topic: Map → Concept → Practice
- production issue: Concept recap → Troubleshooting → Practice
- interview prep: Concept → Interview → Practice
- broad infrastructure topic: Map → Concept → Troubleshooting

## Stop Conditions

Pause and ask a calibration question when:
- the user gives a short acknowledgment but no evidence of understanding
- the topic branches into too many subtopics
- the explanation becomes theory-heavy without operational value
